# TASKS.md — диспетчер очередей по микросервисам

Этот файл — единственная точка входа в очередь. Подробные задачи находятся в каталоге [tasks](tasks/README.md).

## Эталон структуры и роль Structure Keeper

Структура проекта стандартизирована по эталону [_orchestrator_template/](_orchestrator_template/README.md)
(версия структуры — [_orchestrator_template/version.json](_orchestrator_template/version.json)). Системная
роль [Structure Keeper](roles/structure-keeper.md) **запускается первой** — при старте/перезапуске программы
и при создании нового проекта/сервиса — и приводит структуру к эталону: каркас `tasks/`, документацию,
конфигурацию `.orchestrator/`, реестр сервисов и карту зависимостей. Она не пишет код и не выполняет
бизнес-задачи. Глобальное состояние — в [_orchestrator/](_orchestrator/README.md).

Очередь задач ведётся по **папочной модели**: каждая задача — отдельный файл, перемещаемый между стадиями
`tasks/{inbox,ready,in_progress,review,qa,blocked,done,archive}/`. Подробности — [tasks/README.md](tasks/README.md).

## Обязательный алгоритм Claude Code

Claude Code выполняет только роль `PROGRAMMER`: получает от оркестратора уже подготовленную и декомпозированную
задачу одного микросервиса, пишет код и релевантные тесты. Анализ запроса, архитектурная подготовка, pre-coding brief,
разделение по сервисам, декомпозиция, порядок работ и критерии приёмки выполняются оркестратором до передачи задачи.
После Programmer оркестратор выполняет проверки и продолжает цепочку
`SCANNER` → `TASK_REVIEWER` → `PIPELINE_SERVICE` → документационные роли → `GIT_INTEGRATOR`.

1. При старте и после сжатия контекста заново прочитать с диска
   этот файл и [tasks/README.md](tasks/README.md). Нельзя использовать старую копию,
   attachment или summary вместо нового вызова `Read`.
2. Выбирать только свободный блок: запись со `status: "готово к работе"` в `runtime/claude-tasks.json` либо Markdown-очередь `[ ]`.
   Записи с `status: "блок"` или `status: "выполнено"`, а также Markdown-очереди `[B]`, `[R]`, `[!]` и `[x]` обязательно пропускать.
   Поле `service`, scope, зависимости, порядок выполнения и готовый pre-coding brief считаются входным контрактом
   оркестратора. Claude Code не дополняет декомпозицию, не переносит задачу между сервисами и не расширяет scope.
3. **До изменения кода** захватить переданную задачу: поставить runtime-записи `status: "блок"`, а сервисной Markdown-очереди — `[B]`.
   Только после успешной записи этого статуса забирать задачи в работу. Это блокировка владения: другой Claude Code не имеет права повторно взять этот микросервис.
   Переданной задаче поставить `[~]`. Если вход оркестратора неполон, противоречив или требует изменения другого
   сервиса, вернуть `BLOCKED`, не исправляя постановку и не выполняя дополнительную подготовку самостоятельно.
4. Реализовать только заданный scope и добавить или обновить релевантные тесты. Не выполнять тесты, typecheck,
   сборку, lint или иные проверки: их запускают последующие роли оркестратора.
5. Перечислить изменённые файлы и известные по ходу реализации ограничения. Не проводить самостоятельное
   итоговое ревью или проверку критериев приёмки после написания кода.
6. Не выполнять архитектурный анализ, декомпозицию, изменение задач, независимое ревью, pipeline, deploy,
   smoke-проверки, обновление проектной документации, перенос задач в `COMPLETED_TASKS.md`, git commit или push.
7. После написания кода и тестов поставить Markdown-очереди `[R]`. Если задача получена через `runtime/claude-tasks.json`, заполнить у её единственной записи `result`,
   `changedFiles` и `completedAt`, не изменяя `id`, `project` и `service`. **Последним действием** изменить её статус на
   `status: "выполнено"`. Это сигнал Scanner о завершении Programmer; после него не менять код, тесты или поля задачи.

## Требования к постановке задачи и параллельное выполнение

Оркестратор до передачи Programmer определяет затронутые сервисы, создаёт отдельные сервисные задачи, фиксирует
контракты и зависимости, готовит pre-coding brief и критерии приёмки. Каждая передаваемая задача обязана содержать:

- один сервис-владелец и явные границы файлов/компонентов;
- цель и наблюдаемый результат без необходимости дополнительного анализа требований;
- готовые архитектурные решения, API/event/DB-контракты и ограничения;
- зависимости и подтверждение, что они выполнены либо не блокируют реализацию;
- требования к коду и перечень тестовых сценариев, которые Programmer должен реализовать;
- команды и критерии последующей проверки, которые запускает оркестратор;
- известные риски, миграции, требования совместимости и блокирующие вопросы;
- для межсервисной инициативы — общий ID и отдельные задачи каждого сервиса.

Claude Code не меняет эту постановку и не разрезает инициативу между сервисами, но самостоятельно распределяет
реализацию выданной сервисной задачи между своими агентами. Он обязан назначать агентам непересекающиеся области
файлов, учитывать зависимости, собрать изменения и дождаться завершения всех агентов перед сигналом Scanner.
Переходы между ролями, все проверки и git-интеграцию выполняет оркестратор.

## Статусы

Стадия задачи задаётся папкой в `tasks/` и полем `status` во frontmatter файла:

| Стадия (папка) | `status` | Прежний маркер | Значение |
|---|---|---|---|
| `inbox/` | inbox | — | сырые задачи до триажа |
| `ready/` | ready | `[ ]` | готово к работе; Programmer забирает отсюда |
| `in_progress/` | in_progress | `[B]` / `[~]` | сервис захвачен, задача выполняется |
| `review/` | review | `[R]` | код готов, идёт ревью |
| `qa/` | qa | — | проверка: тесты/pipeline/smoke |
| `blocked/` | blocked | `[!]` | заблокировано; в теле обязательна причина |
| `done/` | done | `[x]` | завершено и проверено |
| `archive/` | archive | — | история и legacy-постановки |

## Прямая постановка задачи (пока роли до Programmer не работают)

Пока автоматические роли до Programmer (`ARCHITECT`, `DECOMPOSER`) не запущены и задачи пишутся напрямую,
**задачу нужно сразу записывать со статусом, который Programmer забирает в работу** — иначе она не будет
принята и зависнет:

- В папочной очереди — положить файл задачи в `tasks/ready/` со `status: ready` (готово к работе). Не
  использовать другие стадии (`in_progress`, `review`, `blocked`, `done`, `archive`) для задачи, которая
  должна сразу пойти в работу.
- В `runtime/claude-tasks.json` — `status: "готово к работе"`. Запись должна быть валидной и полной:
  `id` (UUID), `project` и `service` (оба уже зарегистрированы в БД — Scanner их не создаёт), `title`, и
  `status: "готово к работе"`. Поля `result: ""`, `changedFiles: []`, `completedAt: null`. В слоте держать
  одну задачу за раз; документ — `{ "version": 1, "tasks": [ ... ] }`.

Статусы `block`/`блок` и `выполнено`/`done`/`completed` Programmer пропускает, поэтому для прямой постановки
их использовать нельзя.

## Очередь задач

Задачи лежат в стадиях `tasks/<стадия>/` отдельными файлами; владелец указан в поле `service` каждого
файла. Активная очередь (после миграции на папочную модель, версия структуры 1.0.0):

| Сервис | Код | Владелец исходников | Зависимости | ready | review | blocked |
|---|---|---|---|---|---|---|
| orchestrator-service | `ORCHESTRATOR` | `orchestrator-service/` | нет | P1.2, P1.3, P1.4, P2.2, P2.3 | P0.1, P1.1, P2.1 | — |
| frontend (SPA) | `FRONTEND` | `src/` → `dist/` | `/api/*` оркестратора | P1.3 | P0.1, P1.1, P1.2, P2.1, P2.2 | — |
| pipeline-runner | `PIPELINE_RUNNER` | `pipeline-runner/` | этапы/статусы orchestrator-service | P1.2, P2.1 | P1.1 | — |
| scanner-service | `SCANNER` | `scanner-service/` | контракт этапов/Scanner orchestrator-service | — | P1.1 | P1.2, P2.1 |
| tester-service | `TESTER` | `tester-service/` | orchestrator-service, pipeline-runner | — | — | — |
| интеграция/выпуск | `INTEGRATION` | межсервисные E2E | сервисы соответствующего E2E | P2.1–P2.4, P3.1 | — | — |

Зависимости между сервисами зафиксированы в [_orchestrator/dependencies.json](_orchestrator/dependencies.json),
реестр сервисов — в [_orchestrator/services_registry.json](_orchestrator/services_registry.json).
`SCANNER` P1.2 и P2.1 заблокированы незавершёнными зависимостями orchestrator P1.2 и P2.3 (причина — в теле
файлов в `tasks/blocked/`). Прежние сервисные очереди сохранены в
[tasks/archive/legacy-service-queues/](tasks/archive/legacy-service-queues/).

## Общие ограничения

- Любая запись или изменение схемы БД требует отдельного явного подтверждения пользователя до запуска.
  Это включает `INSERT`, `UPDATE`, `DELETE`, `UPSERT`, `MERGE`, backfill, seed, migration,
  `CREATE`, `ALTER`, `DROP`, `TRUNCATE` и косвенный запуск приложения/теста с таким побочным эффектом.
- До подтверждения разрешены только гарантированно read-only диагностика и запросы.
- Нельзя терять или перезаписывать существующие пользовательские изменения рабочего дерева.
- Общие файлы и межсервисные контракты изменяются только после проверки всех потребителей.
- Новая задача добавляется в файл того микросервиса, который владеет изменяемым поведением. Сквозная задача
  разбивается на сервисные части и отдельную задачу интеграции.
- Все модальные окна должны закрываться только явным нажатием пользователя на видимую кнопку внутри модального окна
  (например, «Закрыть», «Отмена», «Сохранить» или кнопка-крестик). Закрытие по клику на фон/overlay, клавише `Escape`,
  потере фокуса или другим неявным событиям запрещено. Это правило применяется ко всем существующим и новым модальным окнам.

## Подключение к Postgres

Общая БД оркестратора `orchestrator_db`. Источник истины — `orchestrator-service/backend/src/config.js`
(приоритет: `config/db.settings.json` → переменные окружения → значения по умолчанию). К Postgres
подключается только orchestrator-service; остальные сервисы и frontend ходят через его `/api/*`.

| Параметр | Локально (по умолчанию) | В Docker / окружении |
| -------- | ----------------------- | -------------------- |
| host | `127.0.0.1` | `host.docker.internal` (`PGHOST`) |
| port | `5432` | `5432` (`PGPORT`) |
| user | `postgres` | `${PGUSER:-postgres}` |
| password | `postgres` | `${PGPASSWORD:-postgres}` (не коммитить) |
| database | `orchestrator_db` | `orchestrator_db` (`PGDATABASE`) |
| admin db | `postgres` | `postgres` (`PGADMIN_DB`) — для `CREATE DATABASE` |

```
postgresql://postgres:postgres@127.0.0.1:5432/orchestrator_db
```

## Постановка межсервисной задачи

До передачи задачи Programmer оркестратор обязан дать инициативе общий ID, описать поток обмена и создать:

1. Отдельную implementation-задачу в `tasks/<Service>.md` для каждого затронутого сервиса.
2. Предшествующую задачу у владельца контракта, если меняется общий `/api`/event/схема.
3. Отдельную задачу в `tasks/Integration.md` для проверки совместимости, E2E, ошибок, rollback и наблюдаемости.

Задачи сервисов A и B должны владеть только своими файлами и после фиксации контракта быть готовы к
параллельному выполнению. Один общий implementation-блок для нескольких сервисов создавать запрещено.
