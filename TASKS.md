# TASKS.md — диспетчер очередей по микросервисам

Этот файл — единственная точка входа в очередь. Подробные задачи находятся в каталоге [tasks](tasks/README.md).

## Обязательный алгоритм Claude Code

Claude Code выполняет роль `PROGRAMMER` для всего готового блока одного микросервиса: пишет код и релевантные
тесты, а также запускает доступные локальные проверки, не изменяющие инфраструктуру. После него работу продолжают
`SCANNER` → `TASK_REVIEWER` → `PIPELINE_SERVICE` → документационные роли → `GIT_INTEGRATOR`.

1. При старте и после сжатия контекста заново прочитать с диска
   этот файл и [tasks/README.md](tasks/README.md). Нельзя использовать старую копию,
   attachment или summary вместо нового вызова `Read`.
2. Выбирать только свободный блок: запись со `status: "готово к работе"` в `runtime/claude-tasks.json` либо Markdown-очередь `[ ]`.
   Записи с `status: "блок"` или `status: "выполнено"`, а также Markdown-очереди `[B]`, `[R]`, `[!]` и `[x]` обязательно пропускать.
   После выбора определить по `service` сервисный файл в `tasks/` и взять в работу **весь незавершённый блок** задач
   из файла этого микросервиса. Выполнять его волнами по зависимостям: в каждой волне брать все готовые задачи, а после их завершения — следующую волну.
3. **До чтения и раздачи задач агентам** захватить блок: поставить runtime-записи `status: "блок"`, а сервисной Markdown-очереди — `[B]`.
   Только после успешной записи этого статуса забирать задачи в работу. Это блокировка владения: другой Claude Code не имеет права повторно взять этот микросервис.
   Всем задачам захваченного блока поставить `[~]`. Заблокированные по существу задачи отметить `[!]` с конкретной
   причиной; они не мешают выполнить остальные независимые задачи того же микросервиса, но не позволяют считать весь блок завершённым.
4. До изменения кода для каждой задачи подготовить предусмотренный ею pre-coding brief. Не включать в
   область работы другие микросервисы, кроме явно описанного контракта или зависимости.
5. Реализовать scope всех взятых задач, добавить или обновить релевантные тесты и запустить доступные
   локальные проверки. Точно зафиксировать изменённые файлы, результаты проверок и остаточные риски.
6. Не выполнять независимое ревью, полный pipeline, deploy, smoke-проверки, обновление проектной документации,
   перенос задач в `COMPLETED_TASKS.md`, git commit или push: эти шаги выполняют последующие роли оркестратора.
7. Дождаться завершения всех параллельных агентов, собрать их результаты и проверить целостность общего diff и тестов
   в границах этого микросервиса. Не переводить runtime-задачу в завершённый статус, пока каждая задача блока не завершена. Если хотя одна задача заблокирована или не завершена, вернуть `BLOCKED` и не посылать Scanner completion.
8. Только после завершения всего блока поставить Markdown-очереди `[R]`. Если блок получен через `runtime/claude-tasks.json`, заполнить у его единственной записи `result`,
   `changedFiles` и `completedAt`, не изменяя `id`, `project` и `service`. **Последним действием** изменить её статус на
   `status: "выполнено"`. Это один общий сигнал Scanner на весь блок микросервиса; после него не менять код, тесты или поля задачи.

## Параллельное выполнение

Claude Code должен пытаться выполнять независимые задачи выбранного микросервиса параллельно в нескольких агентах. Главный
агент остаётся координатором блока, раздаёт непересекающиеся области и собирает их результаты. Параллельное выполнение допустимо, только если:

- зависимости каждой сервисной очереди уже завершены;
- области исходников и файлы задач не пересекаются;
- сервисы не изменяют одновременно один контракт `/api`, миграцию, compose-файл или общую карту;
- каждому агенту назначены конкретный сервисный файл, каталоги исходников и критерии готовности.

Подагенты не меняют `runtime/claude-tasks.json` и не посылают сигнал Scanner. Это делает только главный агент после завершения всего блока. Переходы
между ролями и git-интеграцию по-прежнему выполняют последующие роли оркестратора.

## Статусы

- `[ ]` — готово к работе
- `[B]` — сервис атомарно захвачен Claude Code; повторно брать его запрещено
- `[~]` — отдельная задача захваченного сервиса выполняется
- `[R]` — код готов, идёт итоговое сервисное ревью и проверка
- `[!]` — заблокировано; рядом обязательна конкретная причина
- `[x]` — сервисная очередь полностью завершена и закоммичена

## Общие ограничения

- Любая запись или изменение схемы БД требует отдельного явного подтверждения пользователя до запуска.
  Это включает `INSERT`, `UPDATE`, `DELETE`, `UPSERT`, `MERGE`, backfill, seed, migration,
  `CREATE`, `ALTER`, `DROP`, `TRUNCATE` и косвенный запуск приложения/теста с таким побочным эффектом.
- До подтверждения разрешены только гарантированно read-only диагностика и запросы.
- Нельзя терять или перезаписывать существующие пользовательские изменения рабочего дерева.
- Общие файлы и межсервисные контракты изменяются только после проверки всех потребителей.
- Новая задача добавляется в файл того микросервиса, который владеет изменяемым поведением. Сквозная задача
  разбивается на сервисные части и отдельную задачу интеграции.

## Подключение к Postgres

Общая БД оркестратора `orchestrator_db`. Источник истины — `orchestrator-service/backend/src/config.js`
(приоритет: `config/db.settings.json` → переменные окружения → значения по умолчанию). К Postgres
подключается только orchestrator-service; остальные сервисы и frontend ходят через его `/api/*`.

| Параметр | Локально (по умолчанию) | В Docker / окружении |
| -------- | ----------------------- | -------------------- |
| host | `127.0.0.1` | `host.docker.internal` (`PGHOST`) |
| port | `5432` | `5432` (`PGPORT`) |
| user | `postgres` | `${PGUSER:-postgres}` |
| password | `postgres` | `${PGPASSWORD:-postgres}` (не коммитить) |
| database | `orchestrator_db` | `orchestrator_db` (`PGDATABASE`) |
| admin db | `postgres` | `postgres` (`PGADMIN_DB`) — для `CREATE DATABASE` |

```
postgresql://postgres:postgres@127.0.0.1:5432/orchestrator_db
```

## Постановка межсервисной задачи

До начала реализации Claude обязан дать инициативе общий ID, описать поток обмена и создать:

1. Отдельную implementation-задачу в `tasks/<Service>.md` для каждого затронутого сервиса.
2. Предшествующую задачу у владельца контракта, если меняется общий `/api`/event/схема.
3. Отдельную задачу в `tasks/Integration.md` для проверки совместимости, E2E, ошибок, rollback и наблюдаемости.

Задачи сервисов A и B должны владеть только своими файлами и после фиксации контракта быть готовы к
параллельному выполнению. Один общий implementation-блок для нескольких сервисов создавать запрещено.
