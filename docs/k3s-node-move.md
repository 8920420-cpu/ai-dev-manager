# Переезд ноды k3s между площадками: «воткнул и нажал кнопку»

Дата настройки: 2026-07-08. Нода `k3s-1` (node-1a7af6, MAC `a8:a1:59:1a:7a:f6`,
wg-cluster 192.168.5.1) может переезжать между площадками без правки конфигов
ноды, wg и kubeconfig.

## Что предпровижинено (постоянная конфигурация)

У ноды на каждой площадке есть «слот» — одинаковый хвост адреса `.157`:

| Площадка | Роутер | LAN-подсеть | Слот ноды | DNAT 30080/30443 → слот | DHCP-резервация MAC |
|---|---|---|---|---|---|
| база | giga (192.168.1.1) | 192.168.1.0/24 | 192.168.1.157 | → **192.168.1.213** (ВМ ПС-2, постоянный житель базы) | есть (но см. нюанс про 1.200) |
| Каскад | kaskadvrn.keenetic.link | 192.168.2.0/24 | 192.168.2.157 | → 192.168.2.157 | есть |
| ПС/Барикадная | psvrn.keenetic.link | 192.168.3.0/24 | 192.168.3.157 | → 192.168.3.157 | есть |

- DNAT-правила `ip static tcp <WAN> 30080/30443 <слот>` прописаны на всех трёх
  роутерах ПОСТОЯННО. Площадка без ноды просто висит down у балансировщика
  Timeweb (health-check), это штатно. WAN-интерфейсы: giga=`ISP`,
  kaskad=`GigabitEthernet0/Vlan2`, psvrn=`GigabitEthernet1`.
- Нода зарегистрирована на всех роутерах как known host `k3s-node-1`
  с `ip dhcp host a8:a1:59:1a:7a:f6 x.x.x.157`.
- Нода переведена на чистый DHCP (`dhcp-identifier: mac`): netplan
  `50-cloud-init.yaml` (dhcp4) + `60-ps-lan.yaml` (search ps.lan); статика
  `60-static.yaml` удалена, бэкапы — на ноде в
  `/root/net-migration-backup-20260708/`. cloud-init сеть отключена
  (`99-disable-network-config.cfg`).
- DNS ps.lan на ноде: `/etc/systemd/resolved.conf.d/ps-lan.conf` указывает
  ВСЕ ТРИ роутера (192.168.1.1, 192.168.2.1, 192.168.3.1) — локальный роутер
  площадки отвечает сразу, пин на один роутер убран.
- wg-cluster не трогаем: endpoint у ноды — `k3s-2.ps.lan:51830` (имя),
  у принимающей стороны endpoint не прописан (роуминг).
- Бэкенды LB Timeweb (белые IP площадок) чинит вотчдог `lb-sync-backends`
  (Semaphore, шаблон id=5, каждые 10 минут).

## Процедура переезда

1. Выключить ноду, перевезти, воткнуть в LAN новой площадки — адрес слота
   она получит по DHCP сама.
2. Если новая площадка — ПС/Барикадная: открыть site-to-site транзит на psvrn
   (разовая политика; нужен ноде для wg-дозвона до k3s-2 на базе).
3. Semaphore (localhost:4231, проект «Маршрутка») → шаблон
   **«Нода k3s: переезд на площадку (ip host)»** (id=6) → выбрать площадку →
   Run. Плейбук `ansible/playbooks/node-move.yml` обновит `ip host
   k3s-1.ps.lan` на слот на всех трёх Keenetic и сохранит конфиги.
4. Проверка: `kubectl get nodes` (обе Ready), прод-домены отвечают 200,
   в LB оба живых бэкенда UP.

## Нюансы

- **База: DHCP раздаёт 192.168.1.200 (Windows Server), не giga.** Сегодня
  1.200 выдал ноде запрошенный .157 (лизинг закреплён), резервация на giga
  тоже стоит, но авторитетный DHCP базы — 1.200. Если нода вернётся на базу
  после долгого отсутствия и получит не .157 — добавить резервацию
  a8:a1:59:1a:7a:f6 → 192.168.1.157 в Windows DHCP на 1.200.
- **giga в Ansible/Oxidized ходит по 173.16.82.1** (туннельный IP), потому что
  контейнеры Docker Desktop/WSL2 не видят LAN-подсеть хоста 192.168.1.0/24.
- Пароли роутеров: Semaphore environment «keenetic-routers» (id=3),
  переменные `RPASS_GIGA_LAN` / `RPASS_KASKAD` / `RPASS_PSVRN`;
  дубль — `marshrutka/oxidized/router.db` (не в git).
- DNAT giga «на вырост» смотрит на ВМ ПС-2 (.213): она — постоянная нода базы.
  Если на базе снова появится k3s-1, DNAT можно не трогать (NodePort общий).
- Патч `--tls-san` живёт в юните k3s и перезатрётся апгрейдом k3s через
  install-скрипт (задокументировано также в README подсистемы server).
