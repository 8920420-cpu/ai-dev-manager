# Структура задач (эталон)

Очередь задач организована по стадиям-папкам. Каждая задача — отдельный markdown-файл, который
**перемещается между папками** по мере прохождения цепочки ролей. Имя папки = текущая стадия.

## Стадии

| Папка | Статус | Назначение |
|---|---|---|
| `inbox/` | inbox | Сырые задачи до триажа и постановки. |
| `ready/` | ready | Готовы к работе; Programmer забирает отсюда. |
| `in_progress/` | in_progress | Захвачено и выполняется. |
| `review/` | review | Код готов, идёт ревью. |
| `qa/` | qa | Проверка: тесты/pipeline/smoke. |
| `blocked/` | blocked | Заблокировано; в теле обязательна причина. |
| `done/` | done | Завершено и проверено. |
| `archive/` | archive | История и legacy-постановки. |

## Обязательные поля task-файла

Каждый task-файл обязан содержать (в YAML-frontmatter) минимум:

- `id` — стабильный идентификатор задачи;
- `status` — текущая стадия (совпадает с именем папки);
- `service` — код сервиса-владельца;
- `priority` — приоритет (`P0`, `P1`, …);
- и раздел `## Description` в теле.

Structure Keeper добавляет отсутствующие обязательные поля автоматически (со значением-заглушкой),
не выдумывая содержание задачи.

## Формат

См. [TASK.template.md](TASK.template.md). При перемещении задачи между папками поле `status` в
frontmatter обновляется синхронно с папкой.
