# Миграции структуры

Структура проекта версионируется в `.orchestrator/version.json` (`orchestrator_structure_version`).
Эталонная версия — в `_orchestrator_template/version.json`. Если версия сервиса ниже эталонной,
Structure Keeper применяет миграции из этого каталога **по порядку**, повышая версию шаг за шагом.

## Именование

```
<from_version>_to_<to_version>.md
```

Например: `1.0.0_to_1.1.0.md`. Миграции выстраиваются в цепочку по версиям.

## Текущее состояние

- Эталонная версия: **1.0.0**.
- Реально применяемая миграция: `0.0.0_to_1.0.0.md` (bootstrap: проект без структуры → 1.0.0).
- Файлы `1.0.0_to_1.1.0.md`, `1.1.0_to_1.2.0.md`, `1.2.0_to_1.3.0.md` — **примеры формата** будущих
  миграций. Они не применяются, пока эталон остаётся на 1.0.0; они показывают, как описывать переход
  при повышении версии структуры.

## Правила применения (Structure Keeper)

1. Прочитать версию сервиса и эталонную версию.
2. Если версии равны — миграции не нужны.
3. Если версия сервиса ниже — собрать цепочку файлов перехода от версии сервиса до эталонной.
4. Применять шаги по порядку; после каждого успешного шага обновлять `.orchestrator/version.json`
   и дописывать строку в `.orchestrator/migrations.log`.
5. **Если для текущей версии нет файла перехода (путь неизвестен) — остановиться, ничего не менять,
   записать ошибку в `structure_report.md` и вернуть `BLOCKED`.**
6. Миграции структуры затрагивают только каркас (папки, служебные файлы, шаблоны). Они не меняют
   исходный код, бизнес-логику, API и схемы БД.

## Формат файла миграции

Каждый файл описывает: исходную и целевую версию, мотивацию, конкретные структурные изменения
(`ADD`/`MOVE`/`RENAME`/`REMOVE` каркаса), порядок шагов и критерий завершённости.
