# Integration — сквозная интеграция и выпуск

Владелец: главный агент (межсервисные проверки).
Общие правила, статусы, коммиты и порядок выполнения определены в [TASKS.md](../TASKS.md) и [README.md](README.md).

Здесь живут только межсервисные задачи: E2E-сценарии, проверки совместимости контрактов `/api`,
обработка ошибок, rollback и наблюдаемость. Implementation-части остаются в файлах сервисов-владельцев.

## P2 — Integration

### [ ] P2.1 PIPELINE-STAGE-CONFIG-001 — E2E отключения этапов и настройки Scanner

Initiative: `PIPELINE-STAGE-CONFIG-001`
Owner: `Integration` (выполняет оркестратор, Programmer не назначается)
Scope: сквозные contract/E2E fixtures и тесты; product-код сервисов не исправлять из этой задачи.
Dependencies: завершены соответствующие P0.1/P1.1 в orchestrator-service, frontend, pipeline-runner и scanner-service.

Pre-coding brief (готовит оркестратор):

- Зафиксировать версии сервисов, тестовые каталоги/mount mapping, исходные данные и способ очистки без воздействия на пользовательские данные.
- Любая запись в общей БД требует отдельного подтверждения; предпочтителен изолированный disposable test database.

Tasks:

- Поднять совместимые версии сервисов и проверить единый контракт `enabled`, role code `SCANNER`, `scanner.watchDirectory`, `SKIPPED` и код причины пропуска.
- Пройти E2E: создать проект, выбрать папку Scanner, отключить промежуточный этап, сохранить, перечитать проект и запустить pipeline.
- Подтвердить, что отключённый этап не запускает агента/команды, присутствует в истории как `SKIPPED`, а следующий включённый этап получает управление.
- Отключить Scanner и подтвердить остановку watcher и отсутствие completion; включить обратно и подтвердить восстановление сохранённой папки и exactly-once переход к следующему этапу.
- Проверить отрицательные сценарии: Scanner без папки, относительный/несуществующий/недоступный путь, отсутствующий Docker mount, ошибка API, смена папки во время наблюдения.
- Проверить обновление со старыми данными без `enabled`: все существующие этапы остаются включёнными, миграция/normalization не меняет порядок и роли.
- Проверить observability: UI показывает локальную ошибку, API — стабильный машинный код, scanner readiness — недоступность папки, runner summary/log — причину `disabled_by_configuration`.

Acceptance:

- E2E доказывает полный поток UI → API/storage → runner/scanner → история запуска без ручного исправления данных.
- Ни один отключённый этап не создаёт побочных эффектов, но остаётся видимым и повторно включаемым.
- Scanner невозможно сохранить включённым без папки; ошибка доступности папки после сохранения диагностируется scanner-service.
- Старые проекты продолжают выполняться без изменения поведения.
- Все contract/E2E-тесты стабильны при повторном запуске и не зависят от порядка тестов.

Orchestrator validation:

- Запустить backend/frontend/pipeline-runner/scanner unit suites, затем этот E2E минимум дважды.
- Сохранить артефакты API, readiness и runner summary; проверить cleanup watcher/process/test data.

### [ ] P2.2 PROJECT-TASK-MONITOR-001 — E2E монитора задач проекта

Initiative: `PROJECT-TASK-MONITOR-001`
Owner: `Integration` (выполняет оркестратор, Programmer не назначается)
Scope: сквозные fixtures и browser/API E2E монитора; product-код вне scope.
Dependencies: завершены P1.1 в orchestrator-service и frontend.

Pre-coding brief (готовит оркестратор):

- Зафиксировать временную шкалу событий, два изолированных projectId, timezone и управляемые clock/network delays.
- Подготовить disposable test data и cleanup; запись в общую БД без отдельного подтверждения запрещена.

Tasks:

- Подготовить два проекта с задачами на разных этапах, включая активную, завершённую, заблокированную и задачу с повторным входом в этап.
- Открыть подраздел первого проекта и сверить строки, агрегаты, этапы и длительности с ответом API; затем переключиться на второй проект во время незавершённого запроса.
- Проверить рост длительности активного этапа, неизменность завершённых длительностей, отображение `null`, ручное и автоматическое обновление.
- Проверить отсутствие межпроектной утечки, прекращение polling после закрытия монитора и корректную работу пагинации/фильтров.

Acceptance:

- E2E доказывает поток БД/история событий → API статистики → таблица выбранного проекта без ручного преобразования данных.
- Этапы и длительности совпадают с зафиксированными временными границами, а быстрые переключения проектов не смешивают ответы.
- Тест стабилен при повторном запуске и не зависит от локального часового пояса браузера.

Orchestrator validation:

- Запустить backend/frontend unit suites и browser E2E дважды с разным порядком проектов.
- Проверить остановку polling по сетевым вызовам и отсутствие данных соседнего projectId в DOM/API.

### [ ] P2.3 SERVICE-POST-PROGRAMMER-CONCURRENCY-001 — E2E параллельных цепочек ролей

Initiative: `SERVICE-POST-PROGRAMMER-CONCURRENCY-001`
Owner: `Integration` (выполняет оркестратор, Programmer не назначается)
Scope: controlled-barrier integration fixtures, события и E2E; product-код сервисов вне scope.
Dependencies: завершены P1.2 в orchestrator-service и scanner-service.

Pre-coding brief (готовит оркестратор):

- Определить измеримый интервал перекрытия A/B, точки барьеров, лимит конкурентности и ожидаемую последовательность событий каждой цепочки.
- Использовать изолированные project/service/block IDs и детерминированные fake host/LLM executors.

Tasks:

- Подготовить микросервис A с задачами 1–3 и микросервис B с задачами 1–2 в одном проекте; завершить Programmer-блоки обоих сервисов с управляемой разницей во времени.
- Проверить, что Scanner независимо фиксирует завершение каждого блока и оркестратор создаёт два разных worker-контекста с разными `chainId`.
- Установить барьеры в ролях после Programmer и доказать временное перекрытие цепочек A/B при сохранении последовательного порядка ролей внутри каждой цепочки.
- Проверить повторный completion, ошибку и таймаут одной цепочки, штатное завершение другой, освобождение блокировок и отсутствие перекрёстных артефактов/событий.
- Проверить настраиваемый лимит конкурентности и постановку лишних цепочек в ожидание без потери или повторного запуска.

Acceptance:

- E2E подтверждает, что роли после Programmer выполняются конкурентно между микросервисами и последовательно внутри одного микросервиса.
- Ошибка, задержка или завершение микросервиса A не блокирует и не завершает цепочку B.
- После терминальных статусов не остаётся активных worker-контекстов и зависших сервисных блокировок; повторный Scanner completion не создаёт побочных эффектов.

Orchestrator validation:

- Запустить unit suites orchestrator/scanner, затем controlled-barrier E2E с лимитами 1 и 2.
- Сверить события/логи по `projectId/serviceId/chainId`, отсутствие дубликатов и cleanup после success/failure/timeout.

### [ ] P2.4 PIPELINE-NON-AI-EXECUTOR-001 — E2E прямого запуска pipeline микросервиса

Initiative: `PIPELINE-NON-AI-EXECUTOR-001`
Owner: `Integration` (выполняет оркестратор, Programmer не назначается)
Scope: безопасные pipeline fixtures и orchestrator↔runner E2E; product-код вне scope.
Dependencies: завершены P1.2 pipeline-runner и P1.3 orchestrator-service.

Pre-coding brief (готовит оркестратор):

- Подготовить два disposable service roots и безопасные команды без deploy/network/изменения инфраструктуры.
- Зафиксировать claim/result DTO, допустимые пути и fake AI connector, любой вызов которого немедленно проваливает тест.

Tasks:

- Подготовить микросервисы A и B с различимыми безопасными pipeline-скриптами и перевести задачу A на этап `PIPELINE_SERVICE`.
- Проверить передачу `projectId/serviceId/name/workingDirectory`, выполнение всех действий только скрипта A и структурированное сохранение результата.
- Установить fake AI connector, падающий при любом вызове, и подтвердить полный проход Pipeline без обращения к нему.
- Проверить успешный переход, ошибку команды, неизвестный serviceId и попытку выхода рабочей директории за project root.

Acceptance:

- E2E проходит при полностью недоступных AI-коннекторах для роли Pipeline; AI-вызовов нет.
- Выполняется только pipeline выбранного микросервиса, а результат детерминированно направляет задачу на следующий этап или failure analysis.

Orchestrator validation:

- Запустить unit suites orchestrator/pipeline-runner и E2E для success/failure/unknown service/path traversal.
- Проверить отсутствие prompt exchange/AI agent run и удалить временные service roots/processes.

## P3 — удаление legacy

### [ ] P3.1 LEGACY-CLEANUP-E2E-001 — сквозная проверка удаления fallback-контрактов

Initiative: `LEGACY-CLEANUP-E2E-001`
Owner: `Integration` (выполняет оркестратор, Programmer не назначается)
Scope: read-only inventory, migration/rollback orchestration после подтверждения и канонические/negative E2E; исправления product-кода оформляются отдельными сервисными задачами.
Dependencies: завершены legacy-задачи frontend, orchestrator-service, pipeline-runner и scanner-service; telemetry подтверждает нулевое legacy-использование.

Pre-coding brief (готовит оркестратор):

- Подготовить полный inventory потребителей, backup/restore команды, критерии rollback и отдельный запрос подтверждения любых записей/миграций БД.
- Зафиксировать канонические контракты и ожидаемые стабильные ошибки каждого удалённого fallback.

Tasks:

- До удаления собрать инвентаризацию реальных проектов, pipeline-конфигов, Scanner env и payload `/invoke`; зафиксировать владельца и план миграции каждого найденного потребителя.
- Выполнить миграции с резервной копией и проверяемым rollback; любые записи в БД запускать только после отдельного подтверждения пользователя.
- Пройти E2E только по каноническим путям: серверные бизнес-данные, `{ user }`, обязательный `enabled`, multi-watcher Scanner без feeder bridge.
- Добавить отрицательные проверки старых контрактов: localStorage не влияет на данные, `{ prompt }`, массив stage и legacy env Scanner отклоняются диагностируемо.
- Удалить feature flags и временную telemetry legacy-использования только после согласованного периода нулевых обращений.

Acceptance:

- Поиск production-кода не находит удалённые fallback-ветки; канонический E2E проходит, старые контракты не активируются молча.
- Есть отчёт миграции и rollback-проверки, подтверждающий отсутствие потери проектов, настроек этапов и назначений ролей.

Orchestrator validation:

- До записи выполнить inventory и migration preview; после отдельного подтверждения — backup, migration, audit и rollback rehearsal.
- Запустить все сервисные unit suites и канонический/negative E2E минимум дважды; сохранить отчёт без секретов.
