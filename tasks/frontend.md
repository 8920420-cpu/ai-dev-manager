# frontend (SPA)

Владелец исходников: `src/` (сборка в `dist/`).
Общие правила, статусы, коммиты и порядок выполнения определены в [TASKS.md](../TASKS.md) и [README.md](README.md).

SPA на Vite + React: UI управления проектами, этапами и ролями. В БД ходит только через `/api/*`
оркестратора, напрямую к Postgres не подключается.

## P0 — frontend

### [ ] P0.1 PIPELINE-STAGE-CONFIG-001 — управление активностью этапа и папкой Scanner

Зависимость: `tasks/orchestrator-service.md` → P0.1 (зафиксированный контракт).

Pre-coding brief:

- Перечислить компоненты и reducer мастера создания/редактирования проекта, локальное/API-хранилище и существующий механизм выбора каталога.
- Зафиксировать поведение браузерного directory picker: отображаемое имя/handle нельзя выдавать за абсолютный системный путь. Если API браузера не раскрывает путь, оставить явный ввод абсолютного пути с вспомогательным выбором там, где это технически возможно.

Tasks:

- Добавить в тип `Stage` поле `enabled`; в состояние нового этапа и все пресеты записывать `true`, а для старых локальных данных применять default `true`.
- В каждой строке раздела «Этапы пайплайна» добавить доступный checkbox «Включён». Снятый checkbox означает «этап будет пропущен».
- Визуально обозначать отключённую строку, не скрывая название, роль, Scanner-папку, drag-and-drop и возможность повторного включения. Не использовать только цвет: добавить текстовый статус «Отключён».
- Добавить reducer/action для переключения этапа и обеспечить сохранение значения при перестановке, переходах между шагами, повторном открытии формы и редактировании проекта.
- Для этапа, назначенного роли с кодом `SCANNER`, показать отдельное обязательное поле «Папка для отслеживания» и кнопку выбора каталога. Поле относится к конкретному этапу, а не к общей папке проекта.
- Не распознавать Scanner по строкам `Scanner` в названии этапа/роли. Источник признака — канонический role code из контракта.
- При включённом Scanner не разрешать переход к сохранению без непустого абсолютного пути. Показывать ошибку рядом с полем и переводить фокус на него. При отключённом Scanner ошибка не блокирует сохранение.
- При снятии checkbox не очищать путь. При повторном включении сразу восстановить значение и повторно применить валидацию.
- Если пользователь меняет роль Scanner на другую, сохранить Scanner-настройку в модели, но скрыть её из обычного UI; при возврате роли восстановить значение. Не отправлять секреты или содержимое каталога.
- Обработать ошибки API по `stageId`: показать серверную ошибку у соответствующего этапа, не закрывать модальное окно и не терять введённые данные.
- Обновить карточку/сводку проекта: количество этапов должно явно различать включённые и отключённые, например «7 этапов, 2 отключены».
- Обеспечить keyboard navigation, связанный `label`, `aria-describedby` для подсказки/ошибки и понятные aria-label с названием этапа.

Acceptance:

- Checkbox присутствует у каждого этапа, по умолчанию включён и корректно работает с клавиатуры.
- Отключённый этап остаётся в списке и payload с `enabled: false`; повторное включение возвращает его настройки.
- У включённого Scanner пустая или относительная папка блокирует сохранение с точной локальной ошибкой; у отключённого Scanner — не блокирует.
- Переименование этапа не влияет на появление поля: оно определяется ролью `SCANNER`.
- Создание, редактирование, закрытие/повторное открытие и перезагрузка приложения не сбрасывают checkbox и папку.
- Тесты reducer покрывают defaults/toggle/role change/reorder; component-тесты — видимость, валидацию, фокус, доступность и серверные ошибки; production build проходит.

## P1 — frontend

### [ ] P1.1 PROJECT-TASK-MONITOR-001 — таблица мониторинга задач проекта

Зависимость: `tasks/orchestrator-service.md` → P1.1 (зафиксированный контракт статистики).

Pre-coding brief:

- Найти компонент списка/подразделов проектов, существующую маршрутизацию и способ получения проектов; зафиксировать, какой стабильный `projectId` передаётся в API.
- Согласовать пользовательские подписи для всех `stageCode` и формат длительности. Не дублировать серверную бизнес-логику определения этапа и временных границ.

Tasks:

- Сделать название/строку подраздела проекта интерактивной: нажатие открывает экран или раскрываемую область «Монитор задач» именно выбранного проекта, не запуская редактирование проекта.
- Загрузить `GET /api/projects/:projectId/task-statistics` и показать сводку: всего задач, активных, завершённых, заблокированных и распределение по этапам.
- Отобразить таблицу минимум с колонками «Задача», «Сервис», «Текущий этап», «Статус», «Время на этапе», «Общее время», «Обновлено».
- Форматировать миллисекунды в читаемый вид без потери существенных единиц; `null` показывать как «Нет данных», а не как `0`. Для активных задач обновлять отображаемую длительность относительно серверного `generatedAt`, не меняя исходные данные ответа.
- Добавить состояния загрузки, пустого проекта, ошибки и повторной загрузки. При смене проекта отменять/игнорировать устаревший запрос, чтобы данные предыдущего проекта не появлялись в новом подразделе.
- Добавить пагинацию, сортировку по этапу/статусу/длительности и фильтр по этапу или статусу; параметры UI должны соответствовать контракту API.
- Обновлять монитор автоматически с умеренным интервалом только пока он открыт и вкладка видима; после закрытия/перехода таймер и запросы должны останавливаться. Предусмотреть кнопку ручного обновления и отметку времени последнего успешного ответа.
- Обеспечить открытие с клавиатуры, видимый focus, `aria-expanded`/`aria-controls` для раскрываемого варианта, заголовки таблицы и доступное объявление обновления без постоянного озвучивания тикающего таймера.
- На узком экране сохранить доступ ко всем полям через горизонтальную прокрутку или адаптивное представление, не удаляя этап и время выполнения.

Acceptance:

- Нажатие и клавиатурная активация подраздела проекта открывают таблицу только его задач; повторное действие предсказуемо закрывает её или возвращает к списку.
- Для каждой строки видны текущий этап, статус, время текущего этапа и общее время; активное время увеличивается, завершённое не меняется, неизвестное показано как «Нет данных».
- Пустое состояние, ошибка API, retry, смена проекта, пагинация, фильтры и автообновление покрыты component-тестами с fake timers и отменой запросов.
- Монитор не запускает лишние запросы после закрытия и не показывает данные одного проекта в другом; production build проходит.

### [ ] P1.2 NAV-CATEGORY-ACCORDION-001 — сворачиваемые категории меню навигации

Pre-coding brief:

- Проверить структуру `NAV_ITEMS` и состояние категорий в `Sidebar`, включая настольный, мобильный и полностью свёрнутый варианты бокового меню.

Tasks:

- Сделать категории меню навигации сворачиваемыми и раскрываемыми по нажатию на заголовок категории.
- Хранить единый идентификатор раскрытой категории вместо независимых флагов для каждой категории.
- Одновременно разрешать раскрытие не более одной категории: при открытии второй категории автоматически сворачивать ранее раскрытую.
- Повторное нажатие на заголовок раскрытой категории должно сворачивать её; допускается состояние, в котором все категории свёрнуты.
- Отображать текущее состояние визуальным индикатором и через `aria-expanded`/`aria-controls`; обеспечить управление с клавиатуры.
- Не ломать переходы по пунктам меню, динамический список проектов и поведение мобильной/полностью свёрнутой боковой панели.

Acceptance:

- При открытии категории все остальные категории автоматически сворачиваются; одновременно раскрытых категорий никогда не бывает больше одной.
- Раскрытая категория сворачивается повторным нажатием, мышью и с клавиатуры.
- Состояние и содержимое категорий корректно отображаются в настольном и мобильном меню; навигация по дочерним пунктам продолжает работать.
- Component-тесты покрывают открытие, повторное сворачивание и переключение между двумя категориями; production build проходит.
