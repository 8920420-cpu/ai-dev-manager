# Очереди задач по микросервисам

Корневой [TASKS.md](../TASKS.md) определяет глобальный приоритет, зависимости и статус сервисных очередей.
Этот каталог хранит подробные задачи. Claude всегда начинает и заканчивает работу чтением корневого файла.

## Файлы

- [orchestrator-service.md](orchestrator-service.md) — оркестратор: REST `/api/*`, проекты/этапы/роли, БД Postgres.
- [frontend.md](frontend.md) — SPA на Vite + React (`src/` → `dist/`).
- [pipeline-runner.md](pipeline-runner.md) — исполнитель пайплайна (`.pipeline.json`, стадии build/deploy/smoke).
- [scanner-service.md](scanner-service.md) — файловый Scanner: папка наблюдения и доставка завершённых задач.
- [tester-service.md](tester-service.md) — сервис прогонов тестов (роль Tester).
- [Integration.md](Integration.md) — межсервисные E2E, проверки совместимости и выпуск.

## Правила сервисного файла

- Здесь находятся только задачи сервиса-владельца; межсервисная проверка находится в `Integration.md`.
- Одна межсервисная инициатива всегда даёт отдельную implementation-задачу для каждого затронутого сервиса.
- Все сервисные части ссылаются на общий ID, но владеют только своими исходниками и тестами.
- Общий контракт фиксируется отдельно до параллельного старта сервисных частей.
- Статусы и приоритеты задач синхронизируются с сервисной записью в корневом `TASKS.md`.
- Оркестратор выполняет анализ, архитектурную подготовку, pre-coding brief, разделение инициативы по сервисам, декомпозицию, зависимости и порядок выполнения до передачи задачи Claude Code.
- Claude Code атомарно меняет статус выданной сервисной задачи на `[B]` / `status: "блок"`; такую задачу нельзя взять повторно.
- Агент сервиса не редактирует другие сервисные файлы и не создаёт git-коммит.
- Claude Code в роли Programmer реализует заданный scope и пишет релевантные тесты. Он не меняет постановку и границы сервисов, но может самостоятельно распределять непересекающиеся части реализации между своими агентами и отвечает за сбор их изменений.
- Проверку diff, запуск тестов/typecheck/build/lint, полный pipeline, документацию, перенос в `COMPLETED_TASKS.md`, обновление корневого статуса и git-интеграцию выполняют последующие роли оркестратора.

## Формат задачи

```
### [ ] P1.1 <короткая цель>

Initiative: <общий ID или NONE>
Owner: <один сервис>
Scope: <разрешённые файлы/компоненты>
Dependencies: <готовые зависимости или NONE>

Pre-coding brief (готовит оркестратор):

- <зафиксированное решение и контракт>
- <ограничения, совместимость и риски>

Tasks:

- <конкретное изменение кода>
- <тестовый сценарий, который нужно реализовать>

Acceptance:

- <измеримый критерий приёмки>

Orchestrator validation:

- <команда проверки, которую Programmer не запускает>
- <последующие review/pipeline/E2E действия>
```

Задача не передаётся Programmer, пока отсутствует хотя бы одно обязательное поле, не разрешены блокирующие вопросы
или для изменения другого сервиса не создана отдельная сервисная задача.
