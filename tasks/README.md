# Очередь задач — папочная модель (эталон по умолчанию)

Очередь организована по **стадиям-папкам** согласно эталону `_orchestrator_template/`. Каждая задача —
отдельный markdown-файл, который **перемещается между папками** по мере прохождения цепочки ролей.
Имя папки = текущая стадия и совпадает с полем `status` во frontmatter файла.

Корневой [TASKS.md](../TASKS.md) определяет глобальный приоритет, зависимости и общий алгоритм ролей.

## Стадии

| Папка | Статус | Старый маркер | Назначение |
|---|---|---|---|
| `inbox/` | inbox | — | Сырые задачи до триажа и постановки. |
| `ready/` | ready | `[ ]` | Готовы к работе; Programmer забирает отсюда. |
| `in_progress/` | in_progress | `[B]` / `[~]` | Сервис захвачен, задача выполняется. |
| `review/` | review | `[R]` | Код готов, идёт ревью. |
| `qa/` | qa | — | Проверка: тесты/pipeline/smoke. |
| `blocked/` | blocked | `[!]` | Заблокировано; в теле обязателен раздел причины. |
| `done/` | done | `[x]` | Завершено и проверено. |
| `archive/` | archive | — | История и legacy-постановки. |

## Обязательные поля task-файла

YAML-frontmatter обязан содержать: `id`, `status`, `service`, `priority`, плюс раздел `## Description`
в теле. Полный шаблон — [TASK.template.md](TASK.template.md). Structure Keeper добавляет недостающие
обязательные поля автоматически (со значением-заглушкой), не выдумывая содержание.

## Перемещение между стадиями

При переходе задачи на следующую стадию её файл перемещается в соответствующую папку, а поле `status`
во frontmatter обновляется синхронно. Один файл = одна задача.

## Прямая постановка (пока роли до Programmer не работают)

Задачу пишут сразу в `ready/` со `status: ready` — её забирает Programmer. Для прямой постановки через
слот `runtime/claude-tasks.json` используется `status: "готово к работе"` (см. раздел «Прямая постановка
задачи» в корневом [TASKS.md](../TASKS.md)).

## Архив legacy-постановок

Прежние сервисные очереди (по одному markdown-файлу на сервис со статус-маркерами `[ ]/[R]/[!]`)
перенесены в [archive/legacy-service-queues/](archive/legacy-service-queues/). Их задачи разложены по
стадиям этой модели. Файлы в архиве — только история; новые задачи в них не добавляются.
