# P1.6 PROJECT-TASK-HISTORY-001 — сохранять завершённые задачи и накопительную статистику

## Description

После завершения задачи её запись остаётся в базе данных как часть истории проекта. Показатель «Завершено» в статистике проекта отражает общее количество задач, завершённых за всё время, и никогда не уменьшается из-за очистки активной очереди.

## Scope

- `orchestrator-service/backend/src/db.js`, `taskRunner.js`, `taskStats.js`, обработчики завершения задач и соответствующие backend-тесты.
- Миграция схемы БД допускается только при необходимости и должна сохранять существующие записи.

## Pre-coding brief (готовит оркестратор)

- Каноническая история хранится в PostgreSQL: завершение переводит существующую задачу в терминальный статус `DONE`, фиксирует время завершения и событие перехода, но не удаляет запись.
- Очистка runtime-файла, очереди исполнителя или временных данных не должна удалять каноническую задачу и не должна влиять на накопительную статистику проекта.
- Повторная обработка одного и того же сигнала завершения идемпотентна и не увеличивает счётчик повторно.

## Tasks

- Найти все пути завершения, очистки и повторного запуска задачи; исключить физическое удаление канонической записи задачи при успешном завершении.
- При завершении атомарно перевести задачу в `DONE`, записать `completedAt` и событие перехода статуса.
- Рассчитывать `summary.completed` как количество сохранённых задач проекта в статусе `DONE`, включая задачи прошлых запусков и сессий приложения.
- Разделить очистку активной очереди/временных артефактов и хранение исторических записей в БД.
- Обеспечить идемпотентность повторных completion-сигналов и защиту от двойного учёта одной задачи.
- Добавить тесты завершения, повторного completion, перезапуска процесса и очистки активной очереди.

## Acceptance

- После завершения задача доступна в БД и API статистики со статусом `DONE` и заполненным временем завершения.
- После завершения N разных задач проекта показатель «Завершено» равен N; после перезапуска backend и очистки активной очереди он остаётся равен N.
- Завершение следующей задачи увеличивает показатель до N + 1.
- Повторный сигнал завершения уже завершённой задачи не создаёт дубликат, не меняет историческую запись некорректно и не увеличивает показатель.
- Завершённые задачи одного проекта не влияют на счётчики других проектов.
- Ни один штатный путь успешного завершения задачи не выполняет `DELETE` канонической записи из БД.

## Orchestrator validation

- `npm test` в `orchestrator-service/backend`.
- Интеграционный сценарий: завершить две задачи, перезапустить backend, очистить runtime-очередь и проверить, что обе записи сохранены, а `summary.completed` равен 2.
- Повторно отправить completion для одной задачи и подтвердить, что `summary.completed` остаётся равен 2.

## Result (Programmer)

Реализация без миграции схемы — каноническая история уже хранится в Postgres, а `completedAt`
выводится из `task_events (to_status='DONE')`, поэтому отдельная колонка не нужна.

- Завершение `GIT_INTEGRATOR success → DONE` атомарно (BEGIN/COMMIT) переводит существующую запись и пишет
  событие `TASK_DONE`; физического удаления задачи нет ни на одном пути успешного завершения (проверено
  поиском `DELETE FROM tasks` по `src/` — отсутствует).
- `summary.completed` уже считается как `COUNT(*)` задач проекта в статусе `DONE` по всей БД
  (`getTaskStatistics` в `taskStats.js`) — накопительно по всем сессиям и запускам; очистка runtime-слота
  `tasks/claude-tasks.json` и активной очереди БД не касается.
- Идемпотентность повторного completion: ядро `completeHostTask` вынесено в экспортируемую
  `completeHostTaskTx(c, input)`; lookup задачи переведён с `INNER JOIN` на `LEFT JOIN roles`, и при уже
  терминальном статусе (`DONE`/`CANCELLED`/`FAILED`) функция возвращает `{ duplicate: true }` без события,
  смены статуса и побочных записей — повторный сигнал не плодит дубль и не увеличивает счётчик. Прежний
  `INNER JOIN` давал ложный 404 (у терминальной задачи `current_role_id IS NULL`).

### Изменённые файлы

- `orchestrator-service/backend/src/db.js` — `completeHostTask` → тонкая обёртка над новой
  `completeHostTaskTx`; идемпотентность терминального completion; `LEFT JOIN roles`; `TERMINAL_TASK_STATUSES`.
- `orchestrator-service/backend/test/taskCompletion.test.js` — новый: DONE-переход + событие, отсутствие
  `DELETE`, идемпотентность повтора (`DONE`/`CANCELLED`/`FAILED`), 404, 409 role_mismatch, путь PIPELINE_SERVICE.

### Известные ограничения

- Кумулятивный `summary.completed` после рестарта/очистки активной очереди структурно гарантирован
  (COUNT по сохранённым `DONE`, без DELETE), но end-to-end интеграционный сценарий из раздела validation
  (рестарт backend + очистка очереди) запускает оркестратор отдельно — модульные тесты покрывают только
  обработчик завершения и его идемпотентность.
