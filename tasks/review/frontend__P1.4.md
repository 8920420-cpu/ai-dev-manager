# P1.4 ROLE-CONFIGURATION-001 — модальное окно роли и управление видимостью

## Description

В разделе «Настройки → Роли» сделать каждую роль открываемой. По нажатию показывать модальное окно с кратким описанием роли, её рабочим промтом и списком подключённых skill-файлов. В списке ролей добавить чекбокс «Скрыть»; скрытая роль отображается неактивной во всех интерфейсах «Этапы пайплайна».

## Scope

- `src/features/settings/`, компоненты настройки этапов пайплайна, `src/api/`, `src/types/`, связанные UI-компоненты, стили и тесты.

## Pre-coding brief (готовит оркестратор)

- Использовать API и семантику `hidden` из ORCHESTRATOR-P1.5; не хранить настройки роли только в браузере.
- Клик по интерактивным элементам строки (чекбокс, select, кнопки) не должен одновременно открывать карточку роли.
- Модальное окно закрывается только явной видимой кнопкой внутри окна согласно общему правилу `TASKS.md`; overlay и `Escape` его не закрывают.
- Подключённые skills показываются списком файлов. Кнопка «Добавить» открывает выбор из серверного списка доступных skills; после выбора файл появляется в списке карточки и сохраняется вместе с ролью. Дубли запрещены, подключённый файл можно удалить из роли без удаления самого файла.
- Скрытая роль не удаляется из этапа: она остаётся на своём месте, отображается серой и недоступной для выбора как активное назначение, пока пользователь не снимет «Скрыть».

## Tasks

- Расширить frontend-типы и API-клиент полями описания, prompt, skills и `hidden`.
- Сделать строку/название роли доступной кнопкой открытия модального окна; обеспечить keyboard navigation и видимый focus.
- В модальном окне реализовать редактирование краткого описания и prompt, список skill-файлов, кнопку «Добавить», выбор доступного skill и удаление привязки.
- Добавить в каждую строку роли чекбокс «Скрыть» и сохранять изменение на сервере с понятными loading/error-состояниями.
- Во всех экранах и редакторах «Этапы пайплайна» отображать скрытые роли серыми и неактивными, не удаляя их из сохранённой конфигурации этапа.
- Добавить component/contract-тесты модального окна, выбора skills, сохранения и отображения скрытых ролей.

## Acceptance

- Нажатие на роль открывает карточку с актуальными описанием, prompt и подключёнными skill-файлами; после перезагрузки сохранённые изменения не теряются.
- «Добавить» показывает доступные skills, выбранный файл появляется в карточке ровно один раз и сохраняется; удаление привязки обновляет карточку, но не удаляет файл на сервере.
- Чекбокс «Скрыть» сохраняет состояние. Во всех разделах «Этапы пайплайна» такая роль видна серой/неактивной и не может быть выбрана как активная до снятия флага.
- Управление доступно с клавиатуры, имеет корректные accessible names, focus не теряется, ошибки API показаны пользователю.
- Модальное окно не закрывается по overlay или `Escape`; production build и тесты проходят.

### Ключевые решения
- Модалка закрывается только кнопкой (крестик/Отмена/Сохранить) — переиспользован общий `Modal` (Escape/overlay не закрывают) + `ConfirmDialog` при «грязной» форме, как в `IntegrationFormModal`.
- `hidden` для этапов wizard вычисляется по каноническому коду роли (устойчиво к переименованию), флаг не отправляется в `toStagePayload` (на сервер уходят только roleCodes) — конфигурация этапа не меняется.
- Чекбокс «Скрыть» в секции сохраняется немедленно (PUT `{ hidden }`), отдельно от модалки.
- Дубли skills блокируются на клиенте (нет в списке «Добавить» + защита в addSkill); контракт всё равно вернёт `422 role_skill_duplicate`.

### Риски / блокеры
- Реальные значения backend (`/api/roles`, `/api/skills`) не проверены вживую — только по зафиксированному контракту `api-roles.md`. Если сервер недоступен, секция показывает error-callout, мастер проекта молча игнорирует сбой видимости.
- `StageRow`/wizard — единственные редакторы «Этапов пайплайна» в текущем `src/` (других экранов этапов нет); если появятся новые, флаг `hidden` нужно прокинуть так же.
- Build/typecheck/тесты по инструкции НЕ запускались.
