## P0 — orchestrator-service

### [R] P0.1 PIPELINE-STAGE-CONFIG-001 — контракт включения этапов и папки Scanner

Зависимости: нет. Эта задача фиксирует контракт до начала работ во frontend, pipeline-runner и scanner-service.

Pre-coding brief:

- Описать текущие модели проекта, роли и этапа, все REST-операции их создания/чтения/изменения и фактическое место хранения конфигурации.
- Выбрать единственный канонический признак Scanner: код роли `SCANNER`. Сравнение по отображаемому названию этапа или роли запрещено.
- До изменения схемы БД запросить отдельное явное подтверждение пользователя согласно правилам корневого `TASKS.md`.

Tasks:

- Расширить сохраняемую модель этапа полем `enabled: boolean`. Для новых этапов значение по умолчанию — `true`; старые записи без поля читаются как включённые.
- Добавить к этапу типизированные настройки Scanner с полем `watchDirectory`. Значение — нормализованный абсолютный путь к каталогу; пустая строка не считается значением.
- Хранить `watchDirectory` отдельно для каждого проекта, без глобального общего пути Scanner: проект ПС указывает собственный каталог `tasks`, проект Оркестратор — свой каталог `tasks`; изменение пути одного проекта не должно менять конфигурацию другого.
- Не удалять отключённый этап из проекта и не менять его позицию: состояние, роль, настройки и порядок должны сохраняться для последующего повторного включения.
- Реализовать серверную валидацию create/update: если этап включён и его назначенная роль имеет код `SCANNER`, `watchDirectory` обязателен. Для отключённого Scanner отсутствие папки допустимо; ранее выбранная папка при отключении не очищается.
- Проверять, что `watchDirectory` синтаксически является абсолютным путём. Проверка существования/доступности каталога выполняется scanner-service, поскольку путь может относиться к другой машине или Docker mount.
- Запретить неоднозначную конфигурацию: если сервис поддерживает только один watcher на проект, вернуть документированную ошибку при нескольких включённых этапах `SCANNER`; если поддерживаются несколько watcher, явно закрепить адресацию по `projectId + stageId`.
- Расширить ответы API так, чтобы frontend, pipeline-runner и scanner-service получали `stage.id`, `enabled`, код назначенной роли и `scanner.watchDirectory` без дополнительных догадок.
- Для ошибок использовать стабильные машинные коды, минимум: `scanner_watch_directory_required`, `scanner_watch_directory_must_be_absolute`, `scanner_stage_conflict`; привязать ошибку к конкретному `stageId`.
- Не принимать от клиента вычисленный статус `SKIPPED`: API хранит только конфигурацию, а статус конкретного запуска формирует исполнитель.
- Обновить API-карту/контракт и добавить примеры включённого, отключённого и Scanner-этапа.

Предлагаемый фрагмент контракта:

```json
{
  "id": "stage-id",
  "name": "Scanner",
  "enabled": true,
  "roleIds": ["scanner-role-id"],
  "roleCodes": ["SCANNER"],
  "scanner": {
    "watchDirectory": "K:\\projects\\my-service"
  }
}
```

Acceptance:

- Create/read/update сохраняют `enabled` и папку Scanner без потери данных и возвращают одинаковый контракт.
- Два проекта могут одновременно использовать разные каталоги `tasks`; API возвращает каждому проекту только его `watchDirectory`, а обновление пути ПС не изменяет путь Оркестратора и наоборот.
- Старый этап без `enabled` после чтения ведёт себя как включённый.
- Включённый Scanner без папки и с относительным путём отклоняется с HTTP 422 и стабильным кодом ошибки; отключённый Scanner без папки сохраняется.
- Отключение и повторное включение этапа не удаляет выбранную папку и остальные настройки.
- Сервер определяет Scanner по коду `SCANNER`, поэтому переименование этапа не отключает валидацию.
- Unit/contract-тесты покрывают defaults, create/update, повторное чтение, ошибки валидации и совместимость старых данных.

- Зафиксировать соответствие внутренних `task_status` пользовательским названиям этапов. Не вычислять этап по отображаемому тексту или имени агента.
- Проверить, можно ли получить все метрики из существующих timestamps. Изменение схемы БД выполнять только после отдельного явного подтверждения пользователя по правилам корневого `TASKS.md`.

Tasks:

- Добавить read-only endpoint `GET /api/projects/:projectId/task-statistics`, возвращающий задачи только указанного проекта.
- Для каждой задачи вернуть стабильный контракт: `id`, `title`, `service`, `status`, `stageCode`, `stageName`, `createdAt`, `stageStartedAt`, `completedAt`, `stageDurationMs` и `totalDurationMs`.
- Для активной задачи вычислять `stageDurationMs` от времени последнего перехода в текущий статус до серверного `generatedAt`, а `totalDurationMs` — от создания задачи до `generatedAt`. Для завершённой задачи оба значения фиксировать по соответствующему завершающему событию. Не использовать время браузера для расчётов.

- Добавить агрегаты `total`, `byStage`, `active`, `completed`, `blocked` и `averageCompletedDurationMs`; перечисление задач сортировать по активности, затем по времени последнего изменения.
- Поддержать пагинацию задач и документировать параметры/границы ответа, чтобы монитор не загружал неограниченную историю.
- Возвращать HTTP 404 для неизвестного проекта и стабильный JSON-код ошибки; endpoint не должен изменять задачи, события или запуски.
- Исключить из ответа prompt, response, payload событий, пути, секреты и другие технические данные, не нужные монитору.

Acceptance:

- Контракт однозначно показывает, какая задача находится на каком этапе и сколько времени занимает текущий этап и весь жизненный цикл.
- Активные длительности растут относительно единого `generatedAt`, завершённые остаются стабильными, отсутствующие данные обозначаются `null`.
- Повторные переходы, блокировка, отмена, завершение и проект без задач покрыты unit/contract-тестами.
- Запрос другого `projectId` не раскрывает задачи соседнего проекта; explain/интеграционный тест подтверждает отсутствие N+1 и приемлемую работу пагинации.

### [ ] P1.2 SERVICE-POST-PROGRAMMER-CONCURRENCY-001 — конкурентные цепочки ролей по микросервисам

Initiative: `SERVICE-POST-PROGRAMMER-CONCURRENCY-001`
Owner: `orchestrator-service`
Scope: `orchestrator-service/backend/src/rolePipeline.js`, `roleEngine.js`, `taskRunner.js`, `db.js`, конфигурация лимита и соответствующие backend-тесты.
Dependencies: контракт completion из `tasks/scanner-service.md` → P1.2; входом является завершённый Programmer-блок конкретного микросервиса.

Pre-coding brief (готовит оркестратор):

- К постановке приложена схема текущего диспетчера ролей от `TASK_REVIEWER` до конечного статуса, включая правила блокировки сервиса и точки запуска host/AI-исполнителей.
- Единица конкурентного выполнения зафиксирована как отдельный worker-контекст на один завершённый блок одного микросервиса, идентифицируемый минимум по `projectId + serviceId + blockId`.

Tasks:

- После сигнала Scanner о завершении всех задач Programmer одного микросервиса запускать для него независимый worker-контекст, последовательно выполняющий оставшиеся роли пайплайна до терминального статуса.
- Не использовать одну глобальную последовательную очередь для разных микросервисов: цепочки микросервисов A и B должны иметь возможность выполняться конкурентно и не ждать завершения друг друга.
- Внутри одного микросервиса не запускать две конкурирующие post-Programmer-цепочки для одного и того же блока; повторный сигнал Scanner обрабатывать идемпотентно.
- Передавать между ролями только контекст соответствующих задач и микросервиса. Запретить смешивание `taskId`, артефактов, блокировок, результатов и событий соседних микросервисов.
- Завершать и освобождать worker-контекст после `DONE`, `BLOCKED`, `FAILED` или `CANCELLED`, а также корректно обрабатывать исключения, таймауты и остановку процесса без зависших блокировок.
- Ограничить общую конкурентность настраиваемым лимитом и применять backpressure, не нарушая независимость и порядок цепочки каждого микросервиса.
- Добавить наблюдаемость по `projectId`, `serviceId`, `chainId`, текущей роли и состоянию worker-контекста; не включать секреты и полный prompt в логи.

Acceptance:

- Если у микросервиса A завершены задачи 1–3, а у B — задачи 1–2, Scanner создаёт две независимые цепочки; роли после Programmer для A и B могут выполняться одновременно.
- В каждой цепочке роли выполняются в установленном порядке, а завершение одной цепочки не закрывает и не изменяет состояние другой.
- Повторный completion не создаёт дубликат цепочки или повторные побочные эффекты.
- После терминального статуса worker-контекст и сервисная блокировка освобождены; ошибки одной цепочки не останавливают другие.
- Тесты с управляемыми барьерами подтверждают реальное перекрытие выполнения A/B, последовательность ролей внутри каждого сервиса, изоляцию контекста, лимит конкурентности и корректное завершение worker-контекстов.

Orchestrator validation:

- `npm test` в `orchestrator-service/backend`.
- Запустить integration P2.3 с управляемыми барьерами и подтвердить перекрытие A/B, идемпотентность completion и освобождение блокировок.
- Проверить логи на наличие `projectId/serviceId/chainId` и отсутствие prompt/секретов.

- Удалить или закрыть устаревшие публичные ветки API настроек PostgreSQL, дополнительных БД и импорта локальных БД после перехода всех потребителей. Обновление API- и database-карт передать документационным ролям оркестратора как `documentation_impact: REQUIRED`.

Acceptance:

- Все доступные проектам БД читаются и изменяются через один API и не различаются как «основные»/«дополнительные».
- CRUD, проверка соединения, redaction секрета, сохранение старого пароля при редактировании, запрет удаления используемого подключения и правило единственной БД покрыты contract/integration-тестами.
- Существующие проекты после подтверждённого перехода ссылаются на корректные подключения; в ответах и логах отсутствуют секреты.

Orchestrator validation:
